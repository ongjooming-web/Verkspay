# 2026-03-21 - Prism Invoice & PDF Debugging Session

## Session Overview
Worked on fixing unresponsive PDF download and email send buttons on Prism invoice detail page. Discovered routing issues and file corruption in git history.

## Key Issues Discovered & Fixed

### Issue 1: API Routes Not Being Found (404)
**Problem:** Frontend calls `/api/invoices/{id}/pdf` but route folder was named `[invoiceId]` instead of `[id]`
**Solution:** 
- Renamed folder from `src/app/api/invoices/[invoiceId]/pdf` to `src/app/api/invoices/[id]/pdf`
- Updated params destructuring: `const { id: invoiceId } = await params`
- Added 404-specific error handling in frontend

### Issue 2: File Corruption in Git History
**Problem:** After adding PDF route at commit `0002b3e`, the invoice detail page file `src/app/invoices/[id]/page.tsx` became empty/corrupted
**Root Cause:** File bracket naming on Windows (`[id]`) can cause issues during git operations
**Solution:** 
- Force-reset to stable commit `302d987` where file was intact
- Force-pushed to remote to fix production

### Issue 3: Error Handling & Logging
**Fixed:**
- Added `errorResponse()` helper in both PDF and email routes
- Early returns on all error paths (no hanging requests)
- Detailed console logging at each step
- 404-specific error detection in frontend

## Technical Decisions Made

1. **Route Path Consistency:** Frontend path `/api/invoices/{id}/pdf` requires route folder `[id]`, not `[invoiceId]`
2. **Error Handling:** All API routes must return Response immediately, no unhandled promises
3. **File Naming:** Avoid complex bracket naming in dynamic routes; use simple `[id]` convention
4. **Recovery Strategy:** When git history corrupts, force-reset to last stable commit is faster than trying to repair
