## Key Design Decisions

1. **Webhook Logging Critical:** Every raw payload logged to webhook_logs BEFORE processing. Never lose data.
2. **Multi-Entity Isolation:** All tables use business_unit_id FK for strict data isolation. Users see only their unit.
3. **RLS Security:** All tables protected. Org admins bypass unit isolation.
4. **Generic Webhook Receiver:** Single /api/webhooks/[source] endpoint handles all channels. Source determines parsing logic.
5. **Price Tier Auto-Population:** When creating invoice for Bintang customer, system auto-selects price based on customer's price_tier, but allows manual override.
6. **No Business Logic Yet:** Phase 1 is migrations + webhook receiver + test harness only. No orders, transactions, GL entries.

## Database Schema Reference

### Key Tables (Day 1)
```
organizations
├── id (UUID PK)
├── name (unique, e.g. "Tria Ventures")

business_units
├── id (UUID PK)
├── organization_id (FK)
├── code (unique, e.g. "TC", "BF")
├── name

users_business_units
├── user_id (FK auth.users)
├── business_unit_id (FK)
├── role (org_admin, unit_admin, unit_staff)

channels
├── business_unit_id (FK)
├── code (unique per unit, e.g. "taichew-pos", "shopee")
├── name

webhook_logs (CRITICAL AUDIT TRAIL)
├── business_unit_id (FK)
├── source (taichew-pos, shopee, lazada, direct)
├── raw_payload (JSONB - entire raw payload)
├── processed (boolean)
├── status (received, processed, failed, duplicate)

customers
├── business_unit_id (FK)
├── price_tier (distributor, wholesaler, horeca, ecommerce) [Bintang]
├── credit_term (net_30, net_60, net_90, cod) [Bintang]
├── credit_limit (numeric) [Bintang]
├── outstanding_balance (numeric)

skus
├── business_unit_id (FK)
├── sku_code (unique per unit)
├── sale_price (Tai Chew)
├── distributor_price, wholesaler_price, horeca_price, ecommerce_price [Bintang]
├── unit (meter, liter, batch, etc.)

inventory
├── business_unit_id (FK)
├── sku_id (FK)
├── movement_type (purchase, sale, adjustment, damage)
├── quantity
```
